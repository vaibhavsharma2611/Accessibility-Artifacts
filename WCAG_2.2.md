## WCAG 2.2 Success Criterias

| Success Criteria | Description | Applicable | Non-Applicable | Example |
| :---: | --- | --- | --- | --- |
| [WCAG 2.4.11 Focus Not Obscured (Minimum)](https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html) [AA] | This criterion requires that when a user interface component receives keyboard focus, the component is not entirely hidden by author-created content. | This criterion applies to any user interface component that can receive keyboard focus. | Except for content opened by the component, such as a drop-down menu or a modal dialog. | <li> A navigation menu with sticky headers and footers that do not overlap the focused menu item. </li> <br> <li> A form with input fields and labels that are not covered by a notification banner or a tooltip. </li> <br> <li> A game with icons and buttons that are not hidden by a lightbox or a semi-opaque overlay. </li>
| [WCAG 2.4.12 Focus Not Obscured (Enhanced)](https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-enhanced.html) [AAA] | This criterion requires that when a user interface component receives keyboard focus, no part of the component is hidden by author-created content. | <li> This criterion applies to any user interface component that can receive keyboard focus, including content opened by the component. </li> <br> <li> This criterion does not allow any degree of obscuring, even if the user can reveal the focused component without advancing the keyboard focus. </li> | | <li> A navigation menu with sticky headers and footers that adjust their position or size to avoid covering the focused menu item. </li> <br> <li> A form with input fields and labels that are not affected by a notification banner or a tooltip, or that close or dismiss them on focus. </li> <br> <li> A game with icons and buttons that are not overlapped by a lightbox or a semi-opaque overlay, or that close or dismiss them on focus. </li> |
| [WCAG 2.4.13 Focus Appearance](https://www.w3.org/WAI/WCAG22/Understanding/focus-appearance.html) [AAA] | This criterion requires that the focus indicator for each user interface component has a contrast ratio of at least 3:1 against the adjacent colors, and has a minimum area of 2 CSS pixels. | This criterion applies to all user interface components that can receive keyboard focus. | It is not applicable where the focus indicator is the text cursor. | <li> A button with a thick blue border around it when focused. </li> <br> <li> A link with a yellow background color when focused. </li> <br> <li> A checkbox with a green outline and a checkmark when focused. </li> |
| [WCAG 2.5.7 Dragging Movements](https://www.w3.org/WAI/WCAG22/Understanding/dragging-movements.html) [AA] | This criterion requires that functionality that can be operated by dragging movements can also be operated by pointer inputs without dragging, unless dragging is essential for the function. | This criterion applies to any functionality that involves moving a pointer over a 2D or 3D plane to perform an action, such as drawing, resizing, scrolling, panning, etc. | | <li> A drawing app that allows users to draw lines by clicking on the start and end points, or by using keyboard shortcuts. </li> <br> <li> A video editor that allows users to trim clips by entering the start and end time, or by using buttons or sliders. </li> <br> <li> A map app that allows users to pan the map by using arrow keys, or by using buttons or a compass. </li> |
| [WCAG 2.5.8 Target Size (Minimum)](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html) [AA] | This criterion requires that targets for pointer inputs have a minimum size of 24 by 24 CSS pixels, and a minimum spacing of 24 CSS pixels from other targets. | This criterion applies to any target that can be activated by a pointer input, such as a mouse, a touch screen, or a stylus. | | <li> A navigation menu with large buttons that are spaced apart from each other and from other elements on the page. </li> <br> <li> A form with input fields and labels that are large enough and have sufficient margins and padding around them. </li> <br> <li> A game with icons and buttons that are big enough and have enough space between them and the edges of the screen. </li> |
| [WCAG 3.2.6 Consistent Help](https://www.w3.org/WAI/WCAG22/Understanding/consistent-help.html) [A] | <li> This criterion requires that if a website offers a help option, such as a phone number, chat service, or FAQ page, which appears on multiple pages, it should always be in the same place. </li> <br> <li> This ensures that people who need help can find it more easily, regardless of which page they are on. </li> | | This criterion does not apply to interface-level help, such as contextual help, spell checkers, or instructional text in a form. | <li> A shopping website that has a help link in the header of every page, which leads to a page with contact information, self-help options, and FAQs. </li> <br> <li> A banking website that has a chat icon in the bottom right corner of every page, which opens a chat window with a customer service representative. </li> <br> <li> A social media website that has a menu icon in the top right corner of every page, which contains a link to a help center with various topics and resources. </li>
| [WCAG 3.3.7 Redundant Entry](https://www.w3.org/WAI/WCAG22/Understanding/redundant-entry.html) [A] | <li> This criterion requires that information previously entered by or provided to the user that is required to be entered again in the same process is either auto-populated or available for the user to select. </li> <br> <li> This criterion aims to reduce the cognitive effort and potential errors involved in entering the same information multiple times. </li> | This criterion applies to any information that is asked for more than once during a process, such as a name, address, phone number, etc. | This criterion does not apply if re-entering the information is essential, such as for security or verification purposes, or if the previously entered information is no longer valid. | <li> A shopping website that allows users to check a box to indicate that their billing address is the same as their shipping address, and automatically fills in the billing address fields accordingly. </li> <br> <li> A banking website that remembers the userâ€™s account number and pre-populates it in the login form, so the user only needs to enter their password. </li> <br> <li> A travel website that lets users select their departure and arrival locations from a list of previously entered or provided destinations, instead of typing them again. </li>
